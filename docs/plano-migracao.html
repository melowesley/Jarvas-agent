📊 Plano de Migração: DeepSeek → Gemini

Estratégia estruturada em 5 fases para migrar a lógica de processamento do DeepSeek para o Gemini, mantendo funcionalidade e adicionando redundância ao Jarvas.

Objetivo Geral

✅ Permitir que o Guarda Gemini execute a mesma lógica processamento do DeepSeek

Benefícios esperados:
  • Redundância: Ambos os guardas disponíveis em produção
  • Flexibilidade: Escolher melhor modelo por caso
  • Eficiência: Otimizar custo/latência por tarefa
  • Resiliência: Fallback automático em caso de falha

💭 Feedback Crítico & Ajustes Recomendados

1️⃣ Problema: "Lógica-Fonte" é Ambígua

⚠️ DeepSeek é um modelo de IA, não código determinístico

O que você realmente quer extrair?

Opção O quê Dificuldade Portabilidade
A: Regras de Negócio if/then, parsers, lógica determinística 🟢 Fácil ⭐⭐⭐
B: Prompts (RECOMENDADO) System prompts, instruções que funcionam bem 🟡 Médio ⭐⭐⭐
C: Padrões de Processamento Parsing, classificação, steps sequenciais 🟠 Complexo ⭐⭐
✅ Recomendação: Comece por B (Prompts) - é mais fácil de versionar e portável

2️⃣ Problema: Métrica "95% Matching" Não é Realista

❌ Gemini e DeepSeek são modelos diferentes
Esperar respostas idênticas é ilusório.

Métricas Realistas:

  • ✅ Acuracidade na tarefa (ex: classificação correta)
  • ✅ Satisfação do usuário (sim/não, NPS)
  • ✅ Tempo de resposta (p50, p95, p99)
  • ✅ Taxa de erro (timeouts, mal-formado)
  • ✅ Custo por token (impacto financeiro)

3️⃣ Problema: Falta de Análise de Custo

DeepSeek é mais barato que Gemini. Precisa de business case.

Análise Necessária:

Custo por 1000 tokens (2026):
  DeepSeek Chat:  $0.14 (input) + $0.56 (output)
  Gemini 2.5:     $0.075 (input) + $0.30 (output)

Volume esperado: 10k requisições/dia
  DeepSeek: ~$420/mês
  Gemini:   ~$225/mês

Delta: +$195/mês (ou -$195 se for fallback inteligente)

✅ Ajustes Implementados

  • Foco em Prompts (não regras/código)
  • Métricas realistas (acuracidade, não matching)
  • Análise de custo/ROI
  • Timeline mais agressiva (semana 1: testes rápidos)
  • Dashboard essencial (sucesso, latência, custo)

Artefato 2: Limitações Conhecidas

Rate Limiting Gemini ▼

Gemini tem limites de requisições por minuto. Precisa de backoff exponencial.

Limite: 60 req/min (free tier)
           600 req/min (paid)

Estratégia:
- Queue de requisições
- Backoff exponencial (1s, 2s, 4s, 8s)
- Fallback para DeepSeek se timeout
Diferenças em Parsing de JSON ▼

Gemini às vezes retorna JSON mal-formado. Precisa de validação.

Solução:
1. Usar JSON mode (se disponível)
2. Validar resposta com schema
3. Retry com instrução mais clara
4. Fallback para parsing manual

Passo 1: Extrair Interface Comum

# common/interface.py

from abc import ABC, abstractmethod
from typing import Dict, Any

class QueryProcessor(ABC):
    @abstractmethod
    def parse_query(self, input_text: str) -> Dict[str, Any]:
        """Normalizar entrada"""
        pass

    @abstractmethod
    def classify(self, parsed: Dict) -> str:
        """Classificar intent"""
        pass

    @abstractmethod
    def generate(self, classified: Dict) -> str:
        """Gerar resposta"""
        pass

Passo 2: Adaptador Gemini

# gemini_adapter/adapter.py

from common.interface import QueryProcessor

class GeminiAdapter(QueryProcessor):
    def __init__(self, api_key: str):
        self.client = genai.Client(api_key=api_key)

    def parse_query(self, input_text: str) -> Dict:
        """Usa prompt + regex"""
        response = self.client.generate(
            model="gemini-2.5-flash",
            prompt=f"Parse: {input_text}"
        )
        return self._extract_json(response.text)

    def classify(self, parsed: Dict) -> str:
        """Usa clasificador Gemini"""
        prompt = self._build_classify_prompt(parsed)
        response = self.client.generate(prompt)
        return self._normalize_intent(response.text)

    def generate(self, classified: Dict) -> str:
        """Gera resposta com template"""
        prompt = self._build_generate_prompt(classified)
        response = self.client.generate(prompt)
        return response.text

Passo 3: Orquestrador com Fallback

# router/orchestrator.py

class GuardaOrchestrator:
    def __init__(self, deepseek_adapter, gemini_adapter):
        self.ds = deepseek_adapter
        self.gm = gemini_adapter

    async def process(self, query: str,
                      prefer_model: str = "auto") -> Dict:
        """
        Processa query com fallback automático
        prefer_model: "deepseek" | "gemini" | "auto"
        """

        if prefer_model == "auto":
            # Lógica inteligente de roteamento
            model = self._choose_model(query)
        else:
            model = prefer_model

        try:
            if model == "deepseek":
                result = await self.ds.process(query)
            else:
                result = await self.gm.process(query)

            metrics.record_success(model, result["latency"])
            return result

        except Exception as e:
            metrics.record_error(model, str(e))

            # Fallback
            fallback_model = "gemini" if model == "deepseek" else "deepseek"
            try:
                return await self.process(query, prefer_model=fallback_model)
            except:
                return {"error": "Ambos guardas falharam"}

# tests/test_prompt_portability.py

import pytest
from deepseek_adapter import DeepSeekAdapter
from gemini_adapter import GeminiAdapter
from prompts import CATALOG

@pytest.mark.parametrize("prompt", CATALOG)
async def test_prompt_portability(prompt):
    """Testa cada prompt em ambos modelos"""

    ds_result = await deepseek(prompt["name"], prompt["text"])
    gm_result = await gemini(prompt["name"], prompt["text"])

    # Avaliar ambos
    ds_score = evaluate_response(ds_result, prompt["expected"])
    gm_score = evaluate_response(gm_result, prompt["expected"])

    # Relatório
    print(f"{prompt['name']}")
    print(f"  DeepSeek: {ds_score:.2%}")
    print(f"  Gemini:   {gm_score:.2%}")
    print(f"  Delta:    {abs(ds_score - gm_score):.2%}")

    # Alerta se diferença > 10%
    assert abs(ds_score - gm_score) < 0.10, f"Diferença > 10% em {prompt['name']}"

❓ Questões Críticas a Responder

1. Qual é o Objetivo Real?

💰 Se é ECONOMIZAR... ▼

Estratégia:

  • Usar Gemini como fallback (mais barato)
  • Priorizar: classificação simples → Gemini
  • Fallback para DeepSeek se Gemini falhar
  • Cachear respostas (Redis)
  • Usar batch processing para volumes grandes

Impacto: -30% de custo (se ~5% taxa de erro aceita)

⚡ Se é REDUNDÂNCIA... ▼

Estratégia:

  • Ambos em produção simultaneamente
  • Health check contínuo de ambos
  • Fallback automático em caso de timeout
  • Circuit breaker por modelo

Impacto: 99.99% uptime (vs 99% com um)

🎯 Se é FLEXIBILIDADE... ▼

Estratégia:

  • Expor ambos os modelos ao usuário
  • Deixar escolher: /g (Gemini) ou /d (DeepSeek)
  • A/B testing para otimizar por caso

Impacto: Melhor UX (usuário escolhe)

2. Qual é o Escopo Realista?

Escopo MÍNIMO (MVP - 2 semanas):
  • Adaptador Gemini para 5 prompts críticos
  • Fallback automático
  • Métricas básicas
  • A/B test em staging
Escopo COMPLETO (full migration - 6 semanas):
  • Todos os prompts do DeepSeek
  • Load balancing inteligente
  • Caching distribuído
  • Dashboard Grafana completo
  • Documentação + runbooks

3. Qual é o Orçamento?

Custo Operacional (3 meses):

Cenário A: DeepSeek (atual)
  - 10k req/dia × 30 dias = 300k req/mês
  - Custo: ~$1.500/mês

Cenário B: Gemini (fallback)
  - 90% via Gemini: ~$1.200/mês
  - 10% DeepSeek: ~$150/mês
  - Total: ~$1.350/mês
  - Economia: $150/mês (10%)

Cenário C: Ambos (redundância)
  - 50% cada: ~$1.500 + 1.200 = $2.700/mês
  - Premium de redundância: +$1.200/mês
  - Apenas se uptime crítico

Recomendação: Cenário B (Gemini como fallback)
